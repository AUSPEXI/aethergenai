Ablations with Effect Sizes: Proving What Moves the Needle

By Gwylym Owen — 26–38 min read

Executive Summary

Ablations separate signal from noise. AethergenPlatform reports effect sizes with confidence intervals so teams can see which features, recipes, or thresholds actually improve outcomes—at the operating points that matter.

Why Effect Sizes

  • P‑values alone mislead; buyers need magnitude and uncertainty.
  • Decision‑making needs cost/benefit at operating points.
  • Procurement needs reproducible deltas with seeds/configs.

Design

  • Define base configuration and operating point.
  • Vary one factor at a time or use factorial design.
  • Repeat with different seeds; compute CIs.

Reporting

  • Effect size (Δ utility@OP) with CI.
  • Segment deltas; robustness implications.
  • Evidence links to plots and configs.

Visuals

  • Forest plots of effect sizes with CIs.
  • Trade‑off curves at neighboring thresholds.
  • Segment heatmaps for stability impacts.

Governance

  • Promotion gates include ablation checks.
  • Change‑logs reference effect size IDs.
  • Rollback if negative deltas exceed tolerance.

Case Study

In a payments detector, adding device graph motifs delivered +4.1% utility@1%FPR (CI +3.3–5.0%). Removing amount residuals hurt −1.2%. The forest plot and evidence bundle convinced procurement to approve the change with clear impact.

FAQ

Do we need thousands of runs?

No—enough to bound uncertainty; we target practical CIs around OP.

Can we combine factors?

Yes—use factorial designs and interaction terms; report per‑factor and joint effects.

Glossary

  • Effect size: magnitude of change in KPI.
  • CI: confidence interval for the effect.
  • OP: operating point—where decisions are made.

Checklist

  • Base config frozen; OP defined.
  • Factors and ranges listed.
  • Replications run; CIs computed.
  • Evidence plotted; decisions recorded.

Methodology Details

  • Freeze base pipeline; pin seeds and configs.
  • Define OP; align to budget or analyst capacity.
  • List factors (features, recipes, thresholds) with ranges.
  • Run replications; compute Δ and CIs.

Bootstrap Sketch

# pseudocode
for b in 1..B:
  sample = resample(eval_rows)
  metric_b = utility_at_op(sample)
  store(metric_b)
ci = percentile(metric_b, [2.5, 97.5])

Power Considerations

  • Choose B large enough for stable CIs.
  • Ensure evaluation set covers key segments.
  • Consider variance inflation for rare events.

Interaction Effects

  • Two‑way interactions between top factors.
  • Report both main and interaction Δ.

SOP

  1. Define OP and base.
  2. Run ablations (B runs each).
  3. Compute CIs; generate plots; draft decisions.
  4. Review with QA; attach to evidence; promote or iterate.

Extended FAQ

Why not just report AUC?

Because buyers operate at fixed budgets. Effect sizes at OP map to real work.

How do we pick ranges?

Start from domain knowledge and safety bounds; expand only if gates pass.

What if factors interact?

Run factorial designs and report interactions; decide holistically.
